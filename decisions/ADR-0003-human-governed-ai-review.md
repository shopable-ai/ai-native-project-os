# ADR-0003：以人工治理规则驱动独立 AI 自动审核

- 状态：已接受
- 日期：2026-07-11
- 影响：`human_decision`、`approver`、`verifier`、`waiting_approval`、Acceptance Verdict 和动作授权语义

## 背景

原设计把事实批准、内容审核、Evidence 验证、验收裁决和不可逆动作授权混入 `human_decision`、`approver`、`verifier` 等少数字段，容易使普通业务内容进入 `waiting_approval`，形成逐条人工审核队列。这与 AI-first 系统的目标冲突，也无法稳定支持多语言、多项目。

仅把约束写进生成 Prompt 也不够：Prompt 不是独立权威，难以版本化批准、计算 hash、逐规则引用或在规则变化时传播失效；同一生成节点自审还会形成确认偏差。

## 决策

1. 人类提前定义、批准、发布和维护版本化 Markdown 规则集，AI 只能提出修订建议或登记规则缺口。
2. 生成节点与 `ai_review` 节点分离。AI reviewer 必须按 active 规则集逐项审核并输出精确 `rule_ref`、Evidence、理由和处置。
3. 审核裁决只使用 `allow/rewrite_required/blocked/rule_gap`；可修复问题进入有界改写和新审核 Run，其他失败安全阻断。
4. 普通内容审核不得使用 `waiting_approval`。人工等待只适用于规则/事实/需求治理、例外、剩余风险接受和高风险动作授权。
5. AI 审核与动作授权互不替代：审核通过不授予权限，人工授权不能覆盖 `blocked/rule_gap`。
6. 机器契约和权威入口使用稳定英文路径；人类维护的规则正文使用中文文件名和中文标题。机器只根据稳定 ID、结构化 scope、版本和 hash 定位，不根据文件名或自然语言关键词分流。
7. L1 只定义通用对象、关系、状态、契约、门禁和模板；项目事实和具体规则正文属于 L2/L3。

## 后果

- 人工从日常逐条内容审核中退出，转为规则 owner 和高风险授权者。
- 每个审核结论可以追溯到已批准规则版本和 Evidence。
- 规则缺失或冲突不会被临时人工批准掩盖，而会形成可统计、可回放的治理缺口。
- 需要新增规则集、AI Review Verdict 和规则缺口对象，并同步状态、验收、声明、模板和检查器。
- 当前改动只能证明设计与静态契约成立；在 AI reviewer 运行器、真实模型指纹和 Run/Evidence 出现前，不得声明本地运行或生产能力。

## 被否决方案

1. 只增强生成 Prompt：规则无独立批准、版本和 hash，且生成者自审不独立。
2. 独立 Markdown 规则但仍由同一生成节点自审：缺少职责分离和可信失败证据。
3. 所有审核继续等待人工：吞吐不可扩展，规则缺口无法从日常队列中被系统识别。
4. 用 Approval Ticket 同时批准内容和动作：混淆语义质量与权限边界，可能使高风险授权绕过内容阻断。

## 迁移要求

- 把普通 Workflow 中的人工内容审核节点迁移为独立 `ai_review` 节点。
- 只对规则发布、例外、剩余风险和动作授权保留 `human_decision/waiting_approval`。
- 旧 Verdict/Claim 若未绑定适用规则集和 AI Review Verdict，只能保留历史证据，不能自动升级为新契约下的有效声明。
- L2/L3 逐步把人类可读规则迁入版本化 Markdown 规则包，禁止将业务语言关键词写入 L1 检查器。

## 重审条件

- 异构项目证明同一规则集契约无法表达必要的项目差异；
- 真实运行证明现有 reviewer 独立性不足；
- 规则缺口量长期不可控，说明规则粒度或路由模型需要调整；
- 新的安全或合规要求必须引入不同的强制人工职责。
