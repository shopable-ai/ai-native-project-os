# ADR-0001：采用标准版和自研薄内核

- 状态：已接受
- 日期：2026-07-10
- 决策范围：AI 项目推进操作系统首个设计基线

## 背景

纯文档方法无法强制追溯、门禁、恢复和证据边界；一次建设完整企业平台又会导致过度工程。市面上的规格驱动、智能体运行、研究、架构和观测项目只覆盖局部能力，没有单一方案完整覆盖业务事实到生产证据的链路。

## 选项

1. 纯文档方法论。
2. 一次建设完整企业平台。
3. 标准版、自研薄内核、外部能力适配接入。

## 决策

选择第 3 项：

- 内核只定义六个通用原语；
- 当前启用标准版；
- 多智能体和生产能力按证据升级；
- 所有开源候选先研究、再实验、再锁定；
- 当前业务项目作为存量架构恢复试点，不反向污染通用内核。

## 后果

正面：

- 能先获得可执行价值；
- 避免重复开发成熟基础设施；
- 保留替换运行框架的能力；
- 业务事实、项目数据和通用方法分离。

代价：

- 必须维护适配器和兼容矩阵；
- 第一版仍需自研追溯、门禁、声明和失效传播；
- 在完成本地实验前不能确定最终运行依赖。

## 被否决方案

- 纯文档方法：无法提供机器约束和运行证据。
- 完整企业平台：第一阶段成本和复杂度超过当前需要。
- 直接采用单一 SDD 或智能体框架：无法覆盖业务真相、链路、证据和项目隔离。

## 重审条件

- 出现一个开源项目完整满足薄内核协议并通过迁移实验；
- 标准版治理成本持续超过项目总成本的 20%；
- 两个异构项目证明当前内核无法保持业务中立；
- 生产要求迫使多智能体或持久运行能力成为所有项目的基础依赖。
