# 架构师工作流

架构师职责可以由人类、AI 或人机协作承担，但最终责任人必须是明确的人类角色。

A01—A10 是可以在多个生命周期阶段重复执行的职责，不是框架版本等级、项目治理配置或固定阶段。执行 A08 也不表示已经激活 `multi_agent` 叠加能力。

## 1. 十项架构师工作流

| 编号 | 工作流 | 核心输出 | 完成门禁 |
|---|---|---|---|
| A01 | 项目定位与边界 | 目标、非目标、系统/组织/数据/信任边界 | 范围和责任无重大歧义 |
| A02 | 存量架构恢复 | 当前架构、反向需求、孤儿实现、差距和迁移计划 | 当前状态可以被追溯和解释 |
| A03 | 领域与技术研究 | 来源、未知、候选方案、实验和结论 | 关键假设有权威来源或实验 |
| A04 | 质量属性与风险 | 正确性、安全、恢复、性能、成本等质量场景 | 关键质量属性可测量 |
| A05 | 方案比较与决策 | 买/用/改/自研比较、反方审计、ADR | 决策有背景、证据、后果和重审条件 |
| A06 | 系统结构架构 | 上下文、组件、动态、部署和外部依赖视图 | 每个组件职责和关系明确 |
| A07 | 数据、契约与集成 | 数据所有权、I/O、API/事件、版本和失败语义 | 生产者消费者契约可验证 |
| A08 | AI 原生执行架构 | 模型/代码边界、上下文、智能体、技能、工具、人工规则治理、AI 自动审核和动作授权 | 非确定性步骤有规则引用、评测和安全边界，普通流程不等待人工逐条审核 |
| A09 | 验证与发布架构 | 测试分层、证据目标、发布和回滚方案 | 声明不超过最低证据等级 |
| A10 | 运行与演进架构 | 指标、观测、恢复、成本、迁移和废弃 | 失败、升级和终止均有路径 |

## 2. 存量项目推荐顺序

已有大量实现但前期架构工作不足的 L2/L3 项目，通常优先执行：

```text
A02 存量架构恢复
→ A01 重新确认项目定位
→ A03 补领域与技术研究
→ A04 补质量属性和风险
→ A05 重建关键架构决策
→ A07 校正数据和契约
→ A08 校正 AI 执行边界
→ A09 重建验证和声明基线
```

在上述链路完成前，不应继续大规模扩建新能力。

## 3. 存量架构恢复步骤

1. 冻结当前活动入口和运行状态。
2. 清点需求、文档、代码、契约、测试和运行产物。
3. 生成当前系统上下文和执行链。
4. 反向恢复每项能力的需求、场景和链路来源。
5. 标记无来源实现、无实现需求、无测试契约和无证据声明。
6. 补建缺失研究和架构决策。
7. 建立当前架构、目标架构和差距矩阵。
8. 制定保留、迁移、重做、删除和延后计划。
9. 重建规格、任务和验收基线。
10. 通过架构恢复门禁后再恢复新增功能。

## 4. AI 原生架构特有检查

- 能用确定性代码完成的步骤，不默认交给模型。
- 模型输出必须有结构契约、评测和失败返回。
- 业务规则不得退化为大量自然语言关键词分流。
- 原始资料中的指令不得获得系统权限。
- 智能体、工作流、技能和工具必须分开建模。
- 外部副作用必须有权限、幂等、审批和回执。
- 模型、提示、上下文、工具和策略版本必须可追溯。
- 降级执行必须同步降低完成声明上限。
- 人类批准和维护版本化 Markdown 规则，不逐条审核普通运行内容。
- 普通内容由独立 AI reviewer 产生可追溯 Verdict，并自动允许、改写、阻断或记录规则缺口。
- 内容审核和动作授权正交；审核通过不授予发送、付款、删除或生产发布权限。
