# 框架版本等级模型

本文件是 AI Project OS 自身建设范围的唯一语义权威。框架版本等级回答“L1 产品提供多大范围的通用能力”，不回答某个项目如何治理、处于哪个生命周期阶段或取得什么证据。

`standard` 的中文显示和同名消歧只引用[术语权威](../governance/TERMINOLOGY.md#框架版本等级)；本文只维护框架版本等级的机器语义和能力边界。

## 1. 四级框架

| 等级 | 定位 | 包含能力 | 明确不代表 |
|---|---|---|---|
| `minimal` | 可人工运行的最小薄内核 | 目标、受控对象、基础追溯、最小门禁、Run/Evidence/Claim 基础语义 | 已适合复杂项目或生产运行 |
| `standard` | 当前目标；完整的文件型 AI 原生项目治理 | R0/S0—S7、研究与 ADR、场景链路、契约、状态失效、权限隔离、Run/Evidence/Verdict、最小检查器 | 已实现多智能体平台、持久运行时或企业治理 |
| `platform` | 共享运行和协作平台 | 持久任务、恢复、共享观测、适配器运行、多智能体协调和集中策略服务 | 多租户组织治理或企业资产组合管理 |
| `enterprise` | 最大企业版本 | 多组织/多租户、统一身份、项目组合、合规审计、企业资产目录、集中运营与大规模治理 | 每个项目都必须使用最强治理配置 |

等级按能力范围递增，但不自动证明实现或生产成熟度。一个 `enterprise` 框架可以管理 `lite` 项目；一个 `standard` 框架也可以设计需要 `production` 叠加能力的项目，但在相关能力尚未实现和验证时必须阻断激活。

## 2. 与项目治理配置的边界

项目治理只由项目路由权威定义：基础配置为 `lite | standard`，可叠加能力为 `multi_agent`、`production`。`multi_agent` 描述协作拓扑；`production` 描述生产发布、SLA、监控、恢复和责任链。它们不是 `platform`、`enterprise` 的别名，也不是第三、第四级框架。

## 3. 与成熟度和证据的边界

- 框架等级：目标能力范围。
- `design_status`：规范是否完成设计和审查。
- `implementation_status`：对应实现是否存在并通过实现验收。
- `verification_status`：是否存在绑定范围和环境的 Evidence/Verdict。
- `proof_level`：具体声明所依赖的证据等级。

框架等级不能推导后三者；后三者也不能自动提升框架等级。当前状态只在 `project-os.yaml` 维护，本文件不复制状态值。

## 4. 升级条件

升级框架目标等级必须新增 ADR，列出新增消费者、能力边界、兼容和迁移、成本、安全、运行证据及退出条件。仅因为某个项目采用多个智能体、进入生产或需要某个外部工具，不足以把整个框架升级为 `platform` 或 `enterprise`。
