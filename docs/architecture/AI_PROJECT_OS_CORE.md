# AI 项目推进操作系统核心架构

## 1. 架构决议

采用“`standard` 目标框架版本 + 自研薄内核 + 项目治理配置 + 领域包 + 项目实例 + 开源适配器”。

框架版本等级只由[框架版本等级模型](FRAMEWORK_EDITION_MODEL.md)定义；项目基础治理配置和叠加能力只由[项目类型与治理配置路由](../workflows/PROJECT_TYPE_AND_GOVERNANCE_ROUTING.md)定义。二者不得与生命周期、实现状态或 proof level 混用。

```text
L1 AI Project OS
  → 提供通用方法、协议、状态、门禁、模板和研究方法

L2 业务系统
  → 提供业务事实、场景、链路、契约、工作流和业务技能

L3 项目实例
  → 提供具体项目事实、配置、数据、运行、证据和交付物
```

以上仅为阅读摘要。L1/L2/L3 所有权、依赖方向、`project-os.lock`、兼容、迁移和资产升格只由[仓库与 L1/L2/L3 层级契约](REPOSITORY_AND_LAYER_CONTRACT.md)定义。

## 2. 薄内核七个原语

1. 项目目标与范围。
2. 受控产物及其权威位置。
3. 稳定编号与有类型关系。
4. 状态迁移及重新开启条件。
5. 阶段门禁与完成声明规则。
6. 运行记录与验证证据。
7. 人工治理规则、AI 自动审核与高风险动作授权的职责协议。

以下能力不是标准版强制内核：图数据库、独立注册中心、完整工作流平台、上下文服务、企业资产门户、通用补偿引擎。

## 3. 五个架构平面

| 平面 | 职责 |
|---|---|
| 控制平面 | 阶段、状态、人工批准规则、门禁、评分和声明封顶 |
| 认知平面 | 来源、事实、假设、未知、研究和决策 |
| 设计平面 | 场景、触发、业务链路、能力、功能、架构和契约 |
| 执行平面 | 规格、任务、工作流、技能、工具、独立 AI 审核、检查点和副作用 |
| 证据平面 | 测试、运行结果、规则引用、审核裁决、证据等级、复盘、升格和失效 |

五个平面共享稳定编号和追溯关系，不共享一份无限增长的超级文档。

## 4. 主追溯关系

固定主关系、业务能力与技术能力的分离、Spec 控制外壳和对象关系只由 [受控对象模型](../governance/CONTROLLED_OBJECT_MODEL.md) 定义。R0、S0—S7 只表示成熟度坐标，不是对象链或目录瀑布。

## 5. 框架等级 `standard` 的目标范围

`standard` 框架版本的目标范围必须包含：

- R0、S0—S7 生命周期；
- 领域和技术研究；
- 架构决策记录；
- 场景、触发和链路包；
- 输入输出语义契约；
- 风险自适应规格；
- 任务、工作流和技能边界；
- 追溯检查；
- 分层证据和声明封顶；
- 最小 Run、Evidence、验收裁决和声明协议设计。
- 版本化 Markdown 规则集与 AI 自动审核裁决协议；
- 日常内容审核与高风险外部动作授权的分离。

模拟、本地、只读真实和生产证明是后续实现与验证目标，不是当前设计草案已经实现或验证的能力。

`standard` 框架版本不包含：

- 企业项目组合管理；
- 多租户身份平台；
- 自研持久工作流引擎；
- 自研 AI 可观测平台；
- 全部开源候选的默认安装；
- 未经研究的生产依赖。

## 6. 开源复用边界

外部能力只通过适配器接入。具体候选名称只在 [研究登记表](../../research/registry.yaml) 维护，避免核心架构与研究区形成两份可编辑清单。任何候选都必须完成资料审计、本地实验、退出成本分析、架构决策、适配器契约和一致性证据后才能成为锁定依赖。当前正式第三方运行依赖为空。

## 7. 当前完成度边界

当前设计、实现和验证状态只读取 `project-os.yaml.maturity`；当前评分只读取 `project-os.yaml.scoring_evidence` 指向的快照。本文不复制实时状态或分数。框架目标等级、项目治理选择、实现状态和 proof level 互不推导；设计判断不能代替实现、运行或生产证明。
