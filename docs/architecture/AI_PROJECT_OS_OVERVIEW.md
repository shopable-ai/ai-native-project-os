本文件是人类入口，不维护机器枚举。字段、状态和证据等级始终链接到各自唯一权威；本图出现的文字不能覆盖那些机器定义。

## 1. 四条不可折叠主线

### 1.1 项目推进控制流

回答“项目下一步应该做什么”。从需求种子或存量系统进入类型判断、治理路由和阶段定位，再推进研究、链路、架构、工程、验证与演进。它决定工作顺序，不代替各产物之间的来源关系。

### 1.2 产物追溯依赖链

回答“每个产物为什么存在”。它从来源与批准事实/业务需求，经场景、业务链、能力、功能、功能需求和人类批准基线，再到 ADR、工程链、契约、Spec/Task、Workflow、Skill/Tool、Run、Evidence、Verdict、Claim 和复盘。下游不能绕过上游批准对象自证合理性。

### 1.3 状态坐标

回答“对象现在处于什么状态”。生命周期、工作、审批、实现、证据和失效分别记录；一个 `completed` 不能同时表示阶段已到、实现存在、证据有效和验收通过。

### 1.4 框架和项目配置

回答“系统建设多大、项目采用哪些控制”。框架版本等级描述 L1 能力范围；项目治理配置描述单个项目选择的基础控制和叠加能力。两者互不推导，也不能从阶段、实现或证据等级推导。

四条线不得合并。一个对象可以同时出现在推进流、追溯链、状态坐标和项目配置中，但必须分别记录；任何一条线缺失，都不能用另外三条线补写完成声明。

## 3. 正交坐标怎么读

同一工作项至少要分别回答：当前位于哪个生命周期阶段、工作是否正在进行、实现是否存在、证据达到什么边界、证据是否仍新鲜。中文术语见[术语权威](../governance/TERMINOLOGY.md)，机器状态与迁移见[状态迁移与失效传播](../governance/STATE_TRANSITIONS_AND_INVALIDATION.md)，证据等级见[门禁、证据等级与评分](../governance/GATES_PROOF_SCORING.md)。

## 4. 当前能力边界

当前仓库提供设计规范、静态契约、模板候选和检查器，不是已经运行的项目推进平台。当前总体评分、实现状态和各证明层级只读取 `project-os.yaml`；目标设计分不能外推成当前分，静态测试不能外推成 fixture、本地真实、只读真实或生产证明。
