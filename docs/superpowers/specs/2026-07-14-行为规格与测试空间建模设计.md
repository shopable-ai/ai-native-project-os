## 1. 结论：在 S2 定义行为，在 S4 建模验收空间，在 S5 用稳定案例 ID 实现

本次不新增生命周期阶段，也不把 BDD、ATDD 或 TDD 变成新的项目管理环节。现有主线调整为：

```text
S2：批准 Requirement + 场景/业务链路
    → 父级 Behavior Specification + 稳定 Behavior Case 总表
S3：仅在存在架构选择时产生/更新 ADR
S4：Test Space Modeling + 验收覆盖矩阵 + 最低证明范围/证明上限
S5：Spec / Task / 开发 / BDD、ATDD、TDD 消费稳定 Behavior Case ID
S6：新 Run → Evidence → Verdict → Claim
```

行为案例只描述执行前的用户可观察目标。`actual_result`、`passed`、`evidence_ref` 及任何执行后状态不得进入 Behavior Specification 或 Behavior Case 总表。

## 2. 事实基线：已有链路和测试不归零，缺的是语义到测试空间的系统派生

| 观察面 | 已有能力 | 真正缺口 | 不应重复新建 | 已配置但未系统覆盖 |
|---|---|---|---|---|
| L1 流程 | S2 已有 Requirement 到场景、正常/异常/恢复链路；S4 已有验收判据；S5 已有 Spec/Task；S6 已有 Run/Evidence/Verdict/Claim | S2 没有显式父级 Behavior Specification 和稳定 Behavior Case 总表；S4 没有必需 Test Space Modeling | 不新增阶段，不另建 BDD/ATDD/TDD 生命周期 | 现有场景、acceptance 和失效传播可直接扩展 |
| 验证选择器 | 已是配置驱动、最终回复后置、首文本段精确匹配，并有风险、重复、错误和素材门禁 | 没有“配置空间变化 → 重算验收空间 → 覆盖缺口阻断”Contract | 不重写 selector，不让 UI/provider adapter 实现第二套语义匹配 | 门禁维度多，但完整性仍依赖手写样例 |
| 当前 L2 只读验证案例 | 已有主路径、首段组合边界、后续 segment 反例、复杂/否定、连续同图、错误/素材失败和多接缝测试 | 无配置派生的完整性裁决；项目确定性案例当前已是 13 条，不是旧描述的 5 条，但仍只是局部覆盖 | 不重写现有组件测试 | 当前清单可派生 3 个启用语义类别、10 个 alias、4 个 blocked text；资产级配置只允许 1 个首段组合类别 |
| 派生基础下限 | 项目 manifest/locale 语义清单已可作为批准语义输入 | 缺少可重算公式和失效传播 | 不把当前数量或任何自然语言词面写入 L1 | 当前只读案例的基础下限为 `10 + 30 + 4 + 1 + 3 = 48`；这是配置派生结果，不是 L1 常量 |

## 3. 方案取舍：扩展现有 chain/spec package，不建第二套权威

### 采用：将 S2/S4 执行前产物融入现有 chain package

- `scenarios.md` 承载父级 Behavior Specification、Behavior Case 总表和可选 brownfield As-Is/To-Be 差距表。
- `acceptance.md` 承载 Test Space Model、配置派生规则、验收覆盖矩阵和证明上限，不承载 actual result。
- `chain.yaml` 只新增对上述两个执行前权威的稳定引用，不复制内容。
- spec package 在 S5 必须引用 `behavior_case_refs`，让 Task、acceptance 和 TDD 不再脱离用户目标。

这一方案复用现有阶段、文件、追溯和失效传播，变更小且没有第二套可编辑结论。

### 拒绝：新建“行为差距/行为规格/测试建模”串行阶段

这会把一个 S2 规格活动、一个 S4 验收设计活动和一个 brownfield 可选分析错误地固化为新生命周期。它会与 R0/S2/S4 重叠，并制造阶段状态混用。

### 拒绝：只在测试里增加若干 canonical example

手工选择一两个典型句只能证明这些句子。它不能在 semantic inventory 变化时暴露缺口，也不能证明否定、引用、假设、位置、组合、失败恢复和接缝空间已受控。

## 4. 方法职责分工：行为规格不是 TDD，测试空间不是执行证据

| 方法 | 解决的问题 | 主归属 | 在本框架中的边界 |
|---|---|---|---|
| Behavior Specification | 为什么做、用户应观察到什么、哪些规则和非目标约束行为 | S2 | 父级执行前规格，不记录测试结果 |
| Specification by Example | 用代表性例子消除需求歧义 | S2 | 生成/精炼 Behavior Case，不代替父级规则 |
| Example Mapping | 把规则、例子和未知放在同一审查面 | S2 | 用于找边界和问题，不运行测试 |
| BDD | 让业务行为案例成为实现和可读验收的共同语言 | S5（消费 S2） | 引用稳定 case ID，不反过来批准新业务语义 |
| ATDD | 在开发前把验收约束转成可执行检查 | S4 设计，S5 实例化 | 覆盖矩阵先于实现；执行结果只进 S6 |
| TDD | 以先失败后通过的小步测试驱动代码设计 | S5 | 不负责定义业务目标；测试应回链 Behavior Case |
| As-Is/To-Be Gap Analysis | 在 brownfield/需求变更时比较现状与目标差距 | S2（R0 现状恢复可作输入） | 可选分析，不是通用主阶段，不把现状观察写成验收结果 |
| Test Space Modeling | 从语义清单、规则和边界系统推导需要验收的输入/状态/输出空间 | S4 | 是验收设计，不是 Run 或 Evidence |
| Equivalence Partitioning | 将大量输入分为行为应等价的类 | S4 | 每个有效/无效类都要有代表或 N/A 理由 |
| Boundary Value Analysis | 查找长度、数量、尺寸、位置、顺序和门槛的切换点 | S4 | 边界前/当点/边界后按风险覆盖 |
| Decision Table Testing | 覆盖多个布尔/枚举门禁的决策组合 | S4 | 规则冲突、未知或未授权组合失败关闭 |
| Pairwise/Combinatorial Testing | 用可控成本覆盖多维交互，并在高风险处升级全组合 | S4 | 必须保留维度、约束、组合策略和未覆盖范围 |
| Contrastive/Metamorphic Testing | 验证最小语义变化应/不应改变决策，以及保持语义的变换应保持不变量 | S4 设计，S5 实例化 | AI 可依据批准语义生成候选反例，不得自行批准新语义 |

## 5. S2 模板：一个父级行为规格管理一组稳定案例

### 父级 Behavior Specification 必需字段

| 字段 | 语义 |
|---|---|
| `behavior_spec_id` / `version` | 父级规格稳定 ID 与业务语义版本 |
| `requirement_refs` | 一个或多个已批准 Requirement 引用 |
| `problem_statement` | 当前要解决的问题，不预绑技术方案 |
| `user_observable_goals` | 用户或调用方应能观察到的目标 |
| `behavior_rules` | 稳定行为、不变量和决策边界 |
| `non_goals` | 明确不解决什么 |
| `owner_ref` / `approver_ref` | 责任与业务语义批准主体 |
| `assumptions` / `unknowns` | 当前假设、未知、影响和解决路由 |
| `applicability` | 适用项目类型、环境、locale/channel 或 N/A 边界 |

### Behavior Case 总表必需列

| 字段 | 语义 |
|---|---|
| `case_id` | 稳定 `BC-...` ID，不用 `@v2` 表示实现变化 |
| `requirement_ref` | 本案例覆盖的批准 Requirement |
| `case_type` | `main_path` / `boundary` / `negative` / `failure_recovery` |
| `representative_scenario_or_trigger` | 代表性场景、前置状态或触发，不写技术 fixture 结果 |
| `expected_user_observable_behavior` | 执行前的用户可观察期望 |
| `coverage_target` | 需覆盖的规则、等价类、边界、决策或恢复目标 |

Behavior Case 不含 `actual_result`、`passed`、`evidence_ref`、Run ID 或生产回执。可选 brownfield 差距表只表达 `gap_id`、`behavior_case_ref`、`as_is_observation_ref`、`to_be_target_ref`、`impact`、`decision_route`，它不是新阶段或测试结果表。

### ID 和版本规则

1. 代码、测试、原因假设、实际结果、验证方法或技术方案变化，不改 Behavior Case ID，也不自动升父级版本。
2. 只有批准的用户目标、范围或期望行为变化，才升级父级 Behavior Specification 版本。
3. 新的独立边界、反例或恢复路径优先新建 case ID，而不是复制旧 case 为 `@v2`。
4. 删除或改变语义的 case 使下游进入 `review_required` 或 `invalidated`，历史引用不得静默重绑。

## 6. S4 模板：从批准语义清单派生测试空间，而不是枚举几句话

### 验收覆盖矩阵的 L1 通用字段

| 字段 | 用途 |
|---|---|
| `coverage_id` | 稳定覆盖项 ID |
| `behavior_case_ref` | 对应 S2 稳定 Behavior Case |
| `semantic_inventory_ref` | 已批准的项目语义/等价类清单与版本/hash；无语义清单的项目可指向等价的受控 inventory |
| `inventory_partition_refs_json` / `inventory_member_refs_json` | 本覆盖项实际履行的通用 inventory 分区与成员；成员来自 Test Space 的受控快照，不由 AI 新增语义 |
| `obligation_refs_json` | L2 profile 为这些分区/成员批准的覆盖义务；checker 按 `per_partition` / `per_member` 基数动态闭合 |
| `equivalence_class_or_boundary` | 等价类、边界值、决策表条件或状态维度 |
| `combination_constraints` | 组合允许/排除条件与 pairwise/full-cartesian 选择 |
| `case_relations` | 正例、反例、对比、变形和失败恢复关系 |
| `case_relation` | 可机检的 `positive` / `boundary` / `negative` / `failure_recovery` / `contrastive` / `metamorphic` 关系 |
| `input_boundary` | 受控测试输入或触发边界 |
| `governed_intermediate_boundary` | 完成上游治理、即将进入被测决策节点的中间对象 |
| `expected_transition_or_decision` | 预期状态迁移、决策或失败语义 |
| `observable_output_boundary` | 指定组件、API、Stream、UI、provider 或其他表面的预期可观察输出 |
| `boundary_binding_profile` | 由 L2 将上述四个通用角色绑定到项目对象；L1 只保存占位绑定，不保存业务字段名 |
| `verification_method` | 预定的 unit/contract/component/API/Stream/UI/provider 方法 |
| `minimum_proof_scope` | 这一覆盖项允许的最低证明范围 |
| `coverage_status` | `planned` / `implemented_unverified` / `verification_mapped` / `not_applicable` / `blocked`；只表示覆盖设计/映射状态，不是执行 pass/fail |

`verification_mapped` 只表示已有验证实现与该覆盖项映射；它不等于本次 Run 通过或 Evidence 新鲜。`not_applicable` 必须有结构化理由、owner 和重开条件。

`failure_recovery` 类型的 coverage row 还必须表达 `pre_failure_state`、`failure_terminal`、`recovery_action`、`post_recovery_invariants`、`idempotency_or_side_effect_oracle` 和适用时的 `compensation_ref`；这些仍是事前 oracle，实际恢复结果只进 S6。

### 通用 Contract 只定义 inventory partition 和组合强度

通用 L1 Contract 使用 `inventory_partition_ref`、`dimension_refs`、`dimension_domains_json`、`forbidden_assignments_json`、`constraints`、`generator_kind`、`interaction_strength`、`generation_budget`、`upgrade_triggers` 和 `uncovered_scope` 表达派生，不写 alias、blocked text、selector 或媒体等某个 L2 领域概念。`Derived Combination Registry` 保存执行前的维度 assignment 及 inventory 来源；checker 从 domain 和通用禁配动态重算全部可满足二元组合，再验证 registry 覆盖，不接受固定案例数或仅在测试代码里声明 pairwise。

对当前只读验证 profile，L2 将项目 inventory 绑定为：

- `C` = 启用 semantic category 集合；
- `A` = 全部启用 category 中的批准 alias 集合；
- `B` = 批准 blocked text 集合；
- `L` = 配置允许作为“首段轻反应 + 后续实质回答”的 category 集合；
- `P = {start, middle, end}` = alias 嵌入单一长 segment 的位置；
- `K = {negated, quoted, hypothetical}` = 不得被当作批准语义的上下文类。

最低派生规则：

1. 每个 `C` 至少有一条主路径并能追溯至 Behavior Case。
2. 每个 `A` 有一条独立精确正例，不得只抽 canonical text。
3. 每个 `A` 与 `P`、`K` 由受约束的 strength-2 covering array `CA₂(A,P,K,constraints)` 派生长 segment 反例。Checker 必须验证每个允许的二元组合至少出现一次，不得用固定乘数冒充 pairwise。
4. 每个 `B` 有独立反例。
5. 每个 `L` 有“首段轻反应 + 实质回答”组合正例，后续实质回答不得丢失。
6. 每个 `C` 有 alias 仅位于后续 segment 的反例。
7. 当 `risk_class` 涉及安全/授权/不可逆副作用，已有交互缺陷 Evidence，约束是 unknown/conflicting，或全组合规模不超过已批准 `generation_budget` 时，必须从 pairwise 升级为受约束全组合；否则记录未覆盖三元交互和 claim ceiling。
8. 归一化、重复状态、已有输出、错误/风险/审核、资源缺失/损坏/越界/过大、component/API/Stream/UI/provider 接缝等项目 profile 维度都必须有覆盖约束或明确 N/A。

基础数量下限为：

```text
N_base = |A| + |CA₂(A,P,K,constraints)| + |B| + |L| + |C|
```

这个公式不含第 7/8 类风险扩展的额外案例，因此是下限，不是总数上限。当前只读 L2 配置得到 `|C|=3`、`|A|=10`、`|B|=4`；虽然 3 个类别都使用同一 replacement mode，但实际选择器还要求资产级组合许可，当前只有 `|L|=1`。一个按 alias index 循环偏移 `P/K` 的确定性 covering array 产生 30 行，并由 checker 验证允许的 `A×P`、`A×K`、`P×K` 二元组合全部覆盖，因而当前 `N_base=10+30+4+1+3=48`。L1 只保存 covering-array 规则和验证不变量，不保存这些项目计数。

### 当前只读验证 profile 将通用四边界绑定为项目对象

| 边界 | 必问问题 | 禁止混写 |
|---|---|---|
| `user_input` → `input_boundary` | 用户提出了什么，它是否仅用于触发上游主答 | 不把客户原文当 selector 输入 |
| `governed_reply_segments` → `governed_intermediate_boundary` | 最终治理回复的 segment 边界、顺序和实质回答是什么 | 不把模型草稿或原始 SSE 写成已治理输入 |
| `selector_decision` → `expected_transition_or_decision` | 选择、不选择、回退或阻断的预期是什么 | 不把选择成功当作渲染/投递成功 |
| `rendered_or_delivered_output` → `observable_output_boundary` | 指定组件、API、Stream、UI/provider 表面应观察什么 | 不把本地、UI 模拟、平台接受或生产终端送达互相代替 |

上述四个项目字段只存在于 L2 `boundary_binding_profile` 和其 fixture，不会进入 L1 通用模板、Contract 或 Python/JS 业务分流。

## 7. S5/S6 分层：实现引用案例，实际结果另起 Run

- Spec 记录 `behavior_case_refs` 和它要实现的验收覆盖。
- Task 记录 `behavior_case_refs`、`criterion_ref` 和证据出口，不得从测试文件名倒推需求。
- BDD/ATDD/TDD 测试名或 metadata 引用稳定 case ID；同一 case 可被多个技术表面消费。
- 测试执行生成新 Run；观察值、pass/fail、日志、截图和回执进 Evidence；验收结论进 Verdict；可对外表述进 Claim。
- fixture、local runtime、UI simulation、readonly real、platform accepted、terminal delivered 和 production proof 继续按最低有效 proof level 分层，不得混写。

## 8. 配置漂移与失效：改变语义清单必须重算空间并撤销过度声明

任一以下变化都是 Test Space 失效触发源：

- enabled semantic category 新增、删除、禁用或重命名；
- alias/locale 语义成员新增、删除、修改或归属变化；
- blocked text 新增、删除或修改；
- `replacement_mode`、组合允许策略、业务事实要求或 selector 门禁变化；
- 素材 MIME、尺寸、大小、完整性、路径或安全约束变化。

传播规则：

1. 保留历史 inventory 版本/hash 和旧 Run，不得覆写。
2. 重新计算 `C/A/B/L` 和所有组合维度，生成新的派生清单及差异。
3. 受影响 Behavior Case、coverage row、Spec、Task 和验证实现先标记 `review_required`；期望用户行为已变时，升父级规格版本并使旧下游 `invalidated`。
4. 依赖旧 inventory/hash 的对象先记录独立的 `impact_classification: review_required|invalidated|unaffected`；Evidence 自身只按状态权威把 `stale_status` 更新为 `stale`、`expired` 或 `invalidated`，不得把 `review_required` 写入 Evidence 枚举。对应 Verdict 进入 `pending/revoked`，Claim 进入 `withdrawn/expired`。
5. 只重跑受影响的语义类、维度和表面；停止传播必须有版本隔离或已批准的 `unaffected` 分析。
6. 新 Evidence/Verdict 覆盖全部受影响空间后，才能重新签发 Claim。

## 9. Contract 兼容策略：chain/spec package 升级为 v2，v1 只保留历史可读

独立审计已确认：新内容成为 S2/S4/S5 退出必需项，因此不是 additive extension。`chain-package-contract` 和 `spec-package-contract` 均升级为 major v2，保留原文件名、旧字段语义和 v1 历史可读 profile。

v2 定义四个验证 profile：

| profile | 用途 | 门禁语义 |
|---|---|---|
| `historical_read_v1` | 只解析历史 v1 package | 不允许通过新 S2/S4/S5 退出门 |
| `s2_exit` | 验证 Behavior Specification、Behavior Case registry、owner/approver 和 Requirement 追溯 | 不强迫伪造 S4 覆盖完成状态 |
| `s4_exit` | 继承 `s2_exit`，并验证 Test Space、Coverage、N/A、proof scope 和派生完整性 | 未解析的 inventory/case/coverage 引用 fail-closed |
| `s5_exit` | 继承 `s4_exit`，并验证 Spec/Task/Test 的 `behavior_case_refs` 和 criterion 引用 | 孤儿 Task/测试不得通过 |

`scenarios.md` 和 `acceptance.md` 中的标准 Markdown 表格就是唯一可编辑 registry/matrix，Contract v2 声明精确 section、column、primary key、allowed enum、forbidden field 和 reference integrity；checker 直接解析这些表格，不再新建一份 YAML 业务真相。`chain.yaml` 只保存 `behavior_specification_ref`、`behavior_case_registry_ref`、`test_space_model_ref` 和 `acceptance_coverage_ref`。

profile 调度采用显式、确定性协议，不由 checker 猜测：package 的根 YAML 必须声明 `contract_ref: <contract-id>@<major>`；阶段门禁调用方必须传入 `validation_profile`。`@1` 只允许 `historical_read_v1`，`@2` 禁止使用该历史 profile；`s4_exit` 先执行 `s2_exit`，`s5_exit` 先执行 `s4_exit`，任一前置失败即 fail-closed。`s5_exit` 通过 chain/spec 根引用定位唯一 Markdown registry，再用 canonical `path#section:key` 引用解析 Behavior Case、Coverage、Criterion 和 Task；缺少文件、section、列、主键、引用或 profile/contract 不匹配均返回稳定错误码。控制集若要申请新 S2/S4/S5 门禁，必须引用 chain/spec package contract v2；v1 只能保留历史读取能力。

v1 package 迁移规则：仍可用 `historical_read_v1` 解析，但在新基线下标记 `review_required`；迁移产生新 package revision 并绑定 contract v2；旧 Run/Evidence 不改写，其当前可用性只由影响分析决定。
