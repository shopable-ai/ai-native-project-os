# 受控产物与追溯模型

本文件是各类产物职责、Spec 控制包职责和追溯检查清单的唯一人类可读权威。架构师、Spec 作者和检查器实现者在决定“某项内容应放在哪里、需要追溯到什么、检查器应验证什么”时阅读；读完后应产出职责单一的产物、指向唯一权威的有类型关系和可执行检查项。对象类型、公共字段和关系枚举仍只由 `CONTROLLED_OBJECT_MODEL.md` 定义。

## 1. 规范关系入口

固定主关系、对象类型、公共字段和有类型关系只由 [受控对象模型](CONTROLLED_OBJECT_MODEL.md) 定义。本文件解释产物职责和追溯检查，不维护第二套枚举或对象 Schema。

## 2. 各类产物职责

| 产物 | 回答的问题 |
|---|---|
| 原始来源 | 用户或外部世界提供了什么材料 |
| 业务事实和需求 | 当前批准相信什么、为什么做 |
| 场景和触发 | 谁在什么条件下以什么方式启动 |
| 业务链路 | 业务状态如何一步步变化 |
| 业务能力 | 业务为了达成目标需要哪些稳定能力，不表达技术实现 |
| 功能树 | 用户可以看到和操作什么 |
| 功能需求卡 | 单个功能为什么需要、怎样才算业务正确、边界/异常/验收是什么 |
| 项目地图 | 人类每天从哪里理解目标、阶段、链路、需求、决策、风险和下一步 |
| 需求基线 | 哪些需求版本/hash 已被批准并构成当前验收分母 |
| 上下文快照 | 某次 AI 起草或审查实际包含/排除了哪些权威输入 |
| 技术能力 | 架构决策后形成哪些通用工程能力 |
| 工程链路 | AI、代码、人工和外部系统怎么协作 |
| I/O 契约 | 节点之间如何可靠交接 |
| 规格控制包 | 如何控制范围、计划、任务、验收和追踪 |
| 任务依赖图 | 当前交付具体做什么、先后依赖是什么 |
| 工作流 | 如何有状态、有分支地组织执行 |
| 技能 | 哪个稳定局部能力可被多个工作流复用 |
| 工具 | 哪个确定性执行器产生真实动作 |
| 人工治理规则集 | 人类提前批准哪些 Markdown 约束、适用范围、版本和优先级 |
| AI 审核裁决 | 独立 AI 依据哪些精确规则允许、要求改写、阻断或报告规则缺口 |
| 规则缺口记录 | 哪些运行异常暴露了规则不足，以及谁在何时维护规则 |
| 证据 | 凭什么声称达到某一成熟度 |
| 验收裁决 | 谁基于事前判据和 Evidence 接受、拒绝或有条件接受 |
| 完成声明 | 在指定范围和证据等级内允许表达什么 |

## 3. 一个权威定义，多个有类型依赖

每个受控对象只有一个权威定义位置，但允许多个上游依赖。禁止把“一个权威位置”错误实现成“只能有一个父节点”。关系枚举只在 [受控对象模型](CONTROLLED_OBJECT_MODEL.md) 维护。

## 4. 受控对象字段入口

公共字段和逐类型约束由 [受控对象模型](CONTROLLED_OBJECT_MODEL.md) 唯一定义，包含 owner、executor、approver、verifier、输入输出、门禁、失败路由、失效和替代语义。`approver` 表达规则/事实/需求/范围等治理批准，不表达日常逐条内容审核；AI 审核身份、规则引用和裁决保存在 `ai_review_verdict`。

## 5. 人类需求设计层的位置

功能需求卡是人类第一次理解功能和批准需求的主要入口，不复制 Spec 五件套。每个 Function 只维护一份卡片，分别保存意图、业务过程、边界、风险、验收方向、AI 自检和 Spec 映射。

“已批准实现约束”属于需求边界；“候选实现要点”只属于待研究方案。候选要点经过研究和 ADR 前不得写入 Spec 充当批准要求。项目地图只引用功能需求、链路、决策和风险，不成为第二份需求正文；生成的覆盖/影响视图也不是事实源。

## 6. 规格控制包的位置

规格不是普通线性节点，而是覆盖设计到验收的控制外壳：

| 文件 | 控制职责 |
|---|---|
| `spec.md` | 范围、需求、非目标和失败状态 |
| `plan.md` | 方案、工程落点、顺序和风险 |
| `tasks.md` | 从规格和验收倒推的执行任务 |
| `acceptance.md` | 判据、命令、证据等级和禁止声明 |
| `traceability.md` | 需求到契约、实现、测试和证据的映射 |

规格目录不保存运行代码、脚本、真实数据、运行结果或最终交付物。

## 7. 检查器必须验证

1. 活跃对象是否有稳定编号和唯一权威位置。
2. 权威依赖是否存在非法循环。
3. 原始来源是否绕过批准事实直接驱动实现。
4. P0/P1 功能需求是否具备意图对齐、场景、链路、AI 自检、人类批准、baseline、验收和证据出口。
5. 业务链路节点是否映射到系统、人工或外部责任主体。
6. 工作流是否引用真实存在的 I/O 契约。
7. 技能是否有稳定契约、消费者和失败返回。
8. 任务是否来自规格和验收，而不是凭空新增。
9. 上游变化是否使相关下游证据失效。
10. 完成声明是否超过关键路径最低证据等级。
11. L3 的 Spec、Task、项目 Workflow、Run、Evidence、Verdict 和 Claim 是否绑定当前有效 `route_decision_ref` 与 `control_set_hash`；L1/L2 对象是否避免伪造项目路由引用。
12. active 规则集是否由人类批准、成员 Markdown 引用/hash 是否完整且 scope 匹配。
13. 生成输出是否经过独立 AI 审核、每项 finding 是否引用精确规则、改写是否有界并保存全部尝试。
14. 普通内容审核是否错误进入人工等待队列，或人工动作授权是否覆盖了 `blocked` 审核裁决。
15. Spec 是否精确引用当前 baseline 中已批准的功能需求版本/hash，是否存在 AI 自批或静默改写基线。
16. 上下文快照是否把 `.prompts/`、聊天或临时笔记当作批准事实驱动下游。
