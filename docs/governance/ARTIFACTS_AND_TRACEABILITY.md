# 受控产物与追溯模型

## 1. 产物不是平级清单

```text
需求
→ 场景与触发
→ 业务链路
→ 能力和功能
→ 架构与工程链路
→ I/O 契约
→ 规格控制
→ 任务
→ 工作流
→ 技能和工具
→ 运行
→ 证据
→ 声明
```

## 2. 各类产物职责

| 产物 | 回答的问题 |
|---|---|
| 原始来源 | 用户或外部世界提供了什么材料 |
| 业务事实和需求 | 当前批准相信什么、为什么做 |
| 场景和触发 | 谁在什么条件下以什么方式启动 |
| 业务链路 | 业务状态如何一步步变化 |
| 能力树 | 系统需要哪些稳定能力 |
| 功能树 | 用户可以看到和操作什么 |
| 工程链路 | AI、代码、人工和外部系统怎么协作 |
| I/O 契约 | 节点之间如何可靠交接 |
| 规格控制包 | 如何控制范围、计划、任务、验收和追踪 |
| 任务依赖图 | 当前交付具体做什么、先后依赖是什么 |
| 工作流 | 如何有状态、有分支地组织执行 |
| 技能 | 哪个稳定局部能力可被多个工作流复用 |
| 工具 | 哪个确定性执行器产生真实动作 |
| 证据 | 凭什么声称达到某一成熟度 |

## 3. 一个权威定义，多个有类型依赖

每个受控对象只有一个权威定义位置，但允许多个上游依赖。禁止把“一个权威位置”错误实现成“只能有一个父节点”。

关系类型：

```text
derives_from     来源于
governed_by      受约束于
implements       实现
consumes         消费
produces         产生
verified_by      被验证
invalidates      使失效
supersedes       替代
```

## 4. 受控对象最小字段

```yaml
id: stable-id
type: artifact-type
canonical_path: path/to/authority
version: 1
status: draft
owner_role: architecture
primary_authority: upstream-id
derives_from: []
governed_by: []
implements: []
consumes: []
produces: []
verified_by: []
proof_level: control_package
content_hash: calculated-at-validation
```

## 5. 规格控制包的位置

规格不是普通线性节点，而是覆盖设计到验收的控制外壳：

| 文件 | 控制职责 |
|---|---|
| `spec.md` | 范围、需求、非目标和失败状态 |
| `plan.md` | 方案、工程落点、顺序和风险 |
| `tasks.md` | 从规格和验收倒推的执行任务 |
| `acceptance.md` | 判据、命令、证据等级和禁止声明 |
| `traceability.md` | 需求到契约、实现、测试和证据的映射 |

规格目录不保存运行代码、脚本、真实数据、运行结果或最终交付物。

## 6. 检查器必须验证

1. 活跃对象是否有稳定编号和唯一权威位置。
2. 权威依赖是否存在非法循环。
3. 原始来源是否绕过批准事实直接驱动实现。
4. P0/P1 需求是否具备场景、链路、验收和证据出口。
5. 业务链路节点是否映射到系统、人工或外部责任主体。
6. 工作流是否引用真实存在的 I/O 契约。
7. 技能是否有稳定契约、消费者和失败返回。
8. 任务是否来自规格和验收，而不是凭空新增。
9. 上游变化是否使相关下游证据失效。
10. 完成声明是否超过关键路径最低证据等级。
