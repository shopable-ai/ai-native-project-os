# 能力、功能与 Spec 映射指南

本文件是 `REQUIREMENT_DESIGN_WORKFLOW.md` 的人类操作补充，用于解释复杂业务节点怎样从业务链路展开为能力、功能、功能需求，再进入工程 Spec。它不新增状态轴，也不把具体 L2 业务写入 L1。

## 1. 稳定前缀

| 前缀 | 对象 | 回答的问题 |
|---|---|---|
| `CHAIN-` | 业务链路 | 业务状态怎样从触发走到结果 |
| `CAP-` | 业务能力 | 为完成链路，业务必须具备什么稳定能力 |
| `FUNC-` | 功能 | 使用者或调用方能够观察、调用或验证什么行为 |
| `REQ-` | 需求 | 功能必须满足哪些目标、规则、边界和验收方向 |
| `ADR-` | 架构决策 | 多种实现方案采用哪一种及为什么 |
| `COMP-` | 工程组件 | 工程上由什么组件承担职责 |
| `SPEC-` | 规格包 | 批准需求怎样形成可实施、可测试的工程工作包 |
| `TASK-` | 任务 | 为实现 Spec 具体执行什么工作 |

前缀只帮助稳定引用，不替代中文名称。人类文档标题优先写“中文名称（稳定 ID）”。

## 2. 父级能力与子能力

复杂业务节点可以形成能力树：

```text
CHAIN
→ CAP-root
  → CAP-child-A
  → CAP-child-B
→ FUNC
→ Functional Requirement
→ Spec
```

`CAP-root` 表示一个较大的能力域；子能力仍使用 `business_capability`，不新增模糊的 `subsystem` 业务对象类型。真正的工程子系统在 S3/S4 通过 `component` 与 `engineering_chain` 表达。

## 3. 何时拆 Function

命中任一条件即可考虑拆分：独立业务结果、独立输入输出、独立异常/恢复、独立业务验收、可以被不同调用方单独使用。

不要因为文件太长而拆 Function，也不要把内部判断步骤都升格为顶层 Function。内部判断可以作为某个功能的状态门、规则或子流程。

## 4. 何时拆 Spec

命中任一条件应考虑独立 Spec：独立 owner、独立工程边界、独立发布节奏、独立 proof gate、独立权限或副作用、可以单独替换。

Function 与 Spec 不是强制一一对应：一个 Function 可以由多个 Spec 实现；多个紧密耦合 Function 也可以由一个 Spec 实现，但必须写明拆分理由与验收边界。

## 5. 功能需求卡必须展示执行结果

除目标、规则和边界外，复杂功能卡还应包含：

- 典型输入示例；
- 结构化输出示例或输出契约引用；
- 正例、反例、边界例；
- 业务状态与执行状态的区分；
- 多种候选实现方案；
- 方案进入 Research/ADR 的入口；
- 现有资产与 Spec 映射。

候选实现方案不能冒充批准需求。需求只固定无论采用何种方案都必须满足的结果、边界和验收方向。

## 6. 人类审查顺序

```text
项目地图
→ 父级能力
→ 子能力/功能
→ 功能需求卡与示例
→ 人工批准需求版本
→ ADR/工程设计
→ Spec
```

人类不应先从多个 Spec 五件套反推业务含义；AI 也不得从未批准聊天或候选方案直接生成 active Spec。
